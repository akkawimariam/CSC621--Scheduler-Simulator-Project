Problem description
 1.1 The problem in context
In a database, transactions are units of work that may read and write shared data. When several transactions run at the same time, their operations are interleaved into a single sequence called a schedule (or history). Not every interleaving is safe: some can produce wrong results or make it impossible to recover correctly after a crash. So we need a way to tell whether a given schedule is correct.
Correctness is usually split into two concerns. The first is serializability: the schedule should be equivalent, in some well-defined sense, to running the transactions one after another in some order. The second is recoverability: the order of commits and the way transactions read from each other should not force the system into an inconsistent or unrecoverable state. These ideas are made precise using a precedence (serialization) graph for serializability, and using read-from relationships and commit order for recoverability and related properties (avoiding cascading aborts, strictness, rigor).
The problem addressed in this project is to build a tool that, given a schedule, decides whether it is correct in these senses and explains the answer, i.e., a simulator that performs serializability and correctness analysis and reports results together with reasoning and any violations found.
1.2 What we mean by a schedule and by correctness
Schedules and transactions: A schedule is a total order of operations coming from one or more transactions. Each transaction is assumed to be well-formed: it starts with a START event, ends with exactly one COMMIT or ABORT, and in between performs only data operations (reads and writes on named data items). So the input to the simulator is such a sequence; the simulator may also support generating schedules (e.g., randomly) for testing.
Operations: Besides START, COMMIT, and ABORT, we consider READ, WRITE, INCREMENT, and DECREMENT on data items. INCREMENT and DECREMENT change the value of an item, so for the purpose of deciding who conflicts with whom they are treated as writes: they conflict with any other read or write on the same item.
Serializability: We use conflict equivalence only (not view equivalence). Two operations conflict if they touch the same data item and at least one is a write. From the schedule we build a precedence graph whose nodes are transactions and whose edges reflect “must precede” order implied by conflicting operations. The schedule is conflict-serializable if and only if this graph has no cycle; when it is acyclic, any topological order gives an equivalent serial schedule.
Recoverability and related notions: If a transaction Ti reads a value that was written by another transaction Tj, we say Ti reads from Tj (for that item). Then:

• The schedule is recoverable if whenever Ti reads from Tj and Ti commits, Tj must commit first (so we never commit a reader before the writer it depended on).
• It avoids cascading aborts (ACA) if every read sees only values written by transactions that have already been committed (no “dirty read”).
• It is strict if no one reads or writes an item until the transaction that last wrote that item has either committed or aborted.
• It is rigorous if no one reads or writes an item until the transaction that last read or wrote that item has committed or aborted (a stronger condition than strict).
So the simulator’s job is: given a schedule, compute these notions (precedence graph, read-from, commit/abort positions) and report, for each property, whether the schedule satisfies it, with brief reasoning and any violations (e.g., a cycle, or a read before the writer committed).
Design decisions and assumptions 
2.1 Technology and interface 
The core of the system is implemented in Python 3, chosen for its clear syntax, rich library support for graph algorithms, and ease of extending the analysis logic. All schedule parsing, correctness checks (conflict-serializability, recoverability, ACA, strictness, rigorousness), 2PL/Strict‑2PL validation, and schedule generation run in Python. On top of this core, the project exposes a single web-based user interface. A Flask HTTP API (api.py) wraps the Python parser and scheduler and exposes JSON endpoints for analysis, test‑case retrieval, schedule generation, and 2PL validation. A React (TypeScript, Vite) frontend consumes these endpoints and provides a graphical interface with three main flows: Manual Input (enter transactions and history), Run Test Cases (execute from a built‑in library of test cases in test_cases.py), and Generate Schedule (random interleaving, Basic 2PL, or Strict 2PL). The UI displays detailed step‑by‑step explanations and interactive visualizations of the precedence graph, history diagram, and Strict‑2PL lock history in the browser. This architecture assumes a local deployment where the frontend communicates with the backend over HTTP.
2.2 Input format and well-formedness 
The web interface accepts schedules as space-separated operation tokens over a fixed alphabet. The same token language is used for individual transaction programs and for the global history. For a transaction Ti and data item x, the supported tokens include starti to begin a transaction, ri[x] to read x, wi[x] to write x, inci[x] to increment x, deci[x] to decrement x, ci to commit, and ai to abort.

Data item names are alphanumeric with optional underscores such as x, y, or item_1, as enforced by the parser's regular expressions. In manual mode, the user first provides each transaction in the form Ti: starti … ci or ai, for example T1: start1 r1[x] w1[x] c1, and then provides a single history line such as start1 r1[x] w1[x] start2 r2[y] w2[y] c1 c2.
The parser enforces several well-formedness assumptions for every transaction Ti: the first occurrence of Ti in the schedule must be starti, the subsequence of operations for Ti must end with exactly one ci or ai with no commit or abort appearing earlier, and the history must contain at least one operation, as empty histories are rejected. Violations of these rules, including missing starti, missing ci or ai, or an early commit or abort, result in clear transaction-specific structured error responses from the API.
2.3 Assumptions
• Every schedule provided is complete: each transaction that appears has both its START and its COMMIT or ABORT present in the sequence.
• Serializability is judged only by conflict equivalence; view equivalence is neither used nor assumed.
• Read-from is defined as follows: a read ri(x) is considered to read from the most recent write (or INCREMENT/DECREMENT) on x by a different transaction that has not aborted before the position of that read. Writes by aborted transactions are ignored for read-from.
• When the precedence graph is acyclic, one equivalent serial order is reported (obtained by topological sort).
2.4 Modular architecture
The codebase is organized into clearly separated components that interact through well-defined interfaces. The parsing module (parser.py) performs lexical and syntactic analysis of transaction and schedule strings, enforcing well-formedness requirements such as the presence of a START operation and exactly one commit or abort per transaction. When malformed input is encountered, the parser reports clear validation errors, serving the HTTP API. The data model, implemented across operation.py, transaction.py, and schedule.py, provides the core representations of operations, transactions, and the global schedule history. This module includes conflict predicates and helper functions that are utilized by all subsequent analyses and visualizations throughout the system.
The conflict analysis module (precedence_graph.py) constructs the precedence graph from a given schedule, performs cycle detection to determine conflict-serializability, and computes topological orders to identify serial equivalent schedules. It also renders the graph to DOT and PNG formats for use in the web user interface. The correctness analysis module (scheduler.py) orchestrates all schedule validation checks, including conflict-serializability, recoverability, ACA, strict, and rigorous analyses. This module encapsulates the underlying algorithms and produces structured results that include natural-language explanations and step-by-step reasoning for each validation outcome.
The application layer consists of the backend and frontend. The api.py module is the backend entry point and exposes the core logic through a Flask HTTP API. The test_cases.py module holds the built-in list of test cases used by the API for the Run Test Cases flow. A React and TypeScript frontend in the frontend/ directory consumes the API and provides the graphical user interface and interactive visualizations. Additional modules provide supplementary functionality: schedule_generator.py and schedule_generator_2pl.py for random and protocol-based schedule generation, history_diagram.py for history visualization, and locking.py for 2PL and Strict-2PL validation and lock-history views.
Algorithms used for serializability and correctness checks 
3.1 Conflict-serializability
Conflict-serializability is determined by building a precedence graph from the schedule, checking it for cycles, and, when it is acyclic, deriving an equivalent serial order. The following subsections describe how the precedence graph is built, how cycles are detected, and how a serial order is produced.
3.1.1 Precedence graph construction
The precedence graph has one node per transaction that appears in the schedule. No other nodes are introduced. The edges of the graph represent conflict dependencies: an edge from transaction Ti to transaction Tj means that some operation of Ti precedes and conflicts with some operation of Tj on the same data item. The implementation proceeds by data item. For each data item x, only operations that access x are considered, and they are processed in the order in which they appear in the schedule. Thus the algorithm first groups operations by data item and then, for each such group, scans the operations in schedule order. While scanning, it keeps track of which transactions have already performed a write on x and which have already performed a read on x.
When a read by transaction Tj is processed, a directed edge Ti → Tj is added for every transaction Ti that has already written to x. This captures read–write and write–read conflicts: Tj’s read is constrained to follow those earlier writes. When a write (or an INCREMENT or DECREMENT, which are treated as writes) by Tj is processed, an edge Ti → Tj is added for every transaction Ti that has already read or written x. This captures write–write and read–write conflicts. In this way, an edge Ti → Tj is added exactly when some operation of Ti precedes and conflicts with some operation of Tj on the same data item. Edges are stored in a set so that duplicate edges are not added, and the graph remains a simple directed graph.
Once the precedence graph is constructed, the schedule is conflict-serializable if and only if the graph contains no directed cycles. Cycle detection is performed using depth-first search: if a cycle is found, the schedule is not conflict-serializable, and the cycle path is recorded for explanatory feedback. For acyclic graphs, a topological sort yields a serial equivalent order of transactions.

3.1.2 Serial order
When the precedence graph is acyclic, the schedule is conflict-serializable and there exists at least one topological order of the transactions that corresponds to an equivalent serial schedule. The implementation computes one such order using Kahn’s algorithm. First, the in-degree of each node is computed. A queue is then used to hold all nodes whose in-degree is zero. Nodes are repeatedly removed from the queue; each time a node is removed, it is appended to the topological order, and the in-degrees of all its successors are decremented. If a successor’s in-degree becomes zero, it is added to the queue. The process continues until the queue is empty. The resulting sequence of transactions is reported as one equivalent serial schedule (e.g., T1 → T3 → T2), meaning that executing the transactions in that order would produce the same outcome as the original schedule.

3.2 Recoverability (RC)
A schedule is recoverable if, whenever a transaction Ti reads a data item that was last written by another transaction Tj, then Tj must commit before Ti commits. This property prevents situations where Ti commits based on Tj's updates and Tj later aborts, leaving Ti committed but dependent on uncommitted data. The implementation first computes the read-from relation, then verifies the recoverability rule for every such pair.
3.2.1 Read-from computation
The read-from relation is constructed in a single pass over the schedule. For each transaction Ti reading data item x, the implementation identifies the visible writer: the transaction Tj that performed the most recent write, or increment/decrement operations, on x before this read and that has not aborted before the read's position in the schedule. If such a Tj exists and Tj ≠ Ti, then Ti is considered to read x from Tj, and a read-from tuple (Ti, Tj, x, p) is recorded, where p denotes the position of the read in the schedule. Write operations and increment/decrement operations are treated as updates to x; abort operations are considered to ensure that a read never accesses data written by a transaction that has already aborted at that point in the schedule.

3.2.2 The recoverability rule
Recoverability is defined by a condition on every read-from pair. For each recorded tuple (Ti, Tj, x, p): if Ti commits, then Tj must commit, and the commit of Tj must appear before the commit of Ti in the schedule. Formally, commit(Tj) < commit(Ti) in the schedule order. If Ti never commits or aborts, the pair does not constitute a recoverability violation. This condition ensures that no transaction commits after having read from another transaction that either never commits or commits after it.
3.2.3 Implementation and violation reporting
The implementation of RC utilizes the read-from pairs and the commit position of each transaction, if applicable. For each pair where Ti reads from Tj, the following are checked: that if Ti commits, then Tj commits, and that Tj's commit appears before Ti's commit. If Ti does not commit, the pair is skipped. If Ti commits but Tj never commits, a violation is recorded. If both commit but Tj's commit is not before Ti's, a violation is also recorded. All violations are collected, and explanations are constructed from them, indicating which reader committed after reading from a writer that either never committed or committed too late. The step-by-step explanation enumerates the read-from pairs, states for each whether the rule holds, and provides a summary and conclusion. 

3.3 Avoids cascading aborts (ACA)
A schedule avoids cascading aborts if every time a transaction Ti reads a data item that was last written by another transaction Tj, Tj has already committed before that read. Consequently, no transaction ever reads uncommitted data. This ensures that if a writer later aborts, no reader has already committed based on its writes, eliminating the need for cascading rollbacks. The implementation reuses the read-from relation from the recoverability analysis and verifies the ACA rule on each pair.
3.3.1 The ACA rule
For every read-from pair (Ti, Tj, x, p), the writer Tj must have committed before the read at position p; that is, commit(Tj) < p in the schedule order. Thus, the read by Ti at position p only accesses x after Tj's commit. If for any pair Tj has not yet committed at position p, the schedule does not avoid cascading aborts.
3.3.2 Implementation and violation reporting
The implementation of ACA reuses the read-from pairs and the commit positions. For each pair (Ti, Tj, x, p), it checks whether Tj had committed before position p. If Tj never commits, the rule is violated. If Tj commits at some position, the implementation verifies that this position is before p; if it is not, a violation is recorded. All such violations are gathered, and explanations are constructed accordingly: for each violation, it is noted that Ti read from Tj at position p, but Tj had not yet committed at that time. The step-by-step explanation examines each read-from pair, indicates whether the condition holds, and concludes with a summary and final determination. 

3.4 Strict schedules (ST)
A schedule is strict if no transaction reads or writes a data item x until the transaction that last wrote x has either committed or aborted. So we never read or overwrite data that another transaction has written but not yet finished. The rule applies only when the last writer is a different transaction; a transaction may read or write its own uncommitted data. The implementation checks this with one left-to-right pass over the schedule.
3.4.1 The strictness rule
For each data item x, at every read or write on x, a check determines whether the transaction that last wrote x, if any, has already committed or aborted before this operation. If the last writer is the same as the current transaction, the access is permitted. The first time x is accessed, there is no prior writer, so that access is allowed. Any read or write on x that occurs while another transaction remains the last writer and has not yet committed or aborted constitutes a violation.
3.4.2 Implementation and violation reporting
The strictness algorithm makes one pass over the schedule from left to right. For each data item, the identity of the last writer and its position are kept. The commit position and abort position are recorded for each transaction. At each read or write on data item x by transaction Ti, the last writer of x, designated Tj, is examined. If there is no last writer, the operation is allowed. If the last writer is Ti itself, the operation is allowed. If the last writer is another transaction Tj, a check determines whether Tj had already committed or aborted before this position. If yes, the operation is allowed. If not, a violation is recorded: at this position, Ti is reading or writing x while Tj, the last writer of x, has not yet committed or aborted. After the pass, an explanation is built from the violations, describing at which position which transaction accessed which item while the previous writer had not finished. The step-by-step explanation describes each relevant access, including first access, same-transaction access, or check against the last writer and ends with a summary and conclusion.

3.5 Rigorous schedules
A schedule is rigorous if it obeys a stronger condition than strictness: for reads, the same rule as for strict schedules applies, and for writes, every transaction that previously read or wrote the item must have finished, meaning it has committed or aborted, before the next write. So no transaction writes a data item while another transaction still has an uncommitted read or write on it. The implementation checks both rules in a single left-to-right pass over the schedule.
3.5.1 The rigorous rules
For each read on a data item x, the rule mirrors that of strict schedules: the transaction that last wrote x must have committed or aborted before this read. Therefore, no transaction reads x while another transaction remains the last writer and has not yet finished. For each write on a data item x, no transaction may write x until every transaction that previously read or wrote x has committed or aborted. In other words, the last transaction that read or wrote x must have committed or aborted before this write. Consequently, no transaction overwrites x while another transaction maintains an uncommitted read or write on x. As with strictness, the constraint applies only when the last accessor is a different transaction; a transaction may write x after its own prior read or write without committing in between.
3.5.2 Implementation and violation reporting
The implementation performs a single left-to-right pass over the schedule. For each data item x, two pieces of state are maintained: the identity and position of the last transaction that wrote x, and the identity and position of the last transaction that accessed x, whether read or write. Precomputed information about where each transaction commits and aborts is also utilized.
At each read on x by transaction Ti, the last writer of x, if any, is examined. If the last writer is another transaction Tj, the algorithm verifies whether Tj had already committed or aborted before this read. If not, a violation is recorded. The last access state for x is then updated to reflect this read, recording the position and Ti. At each write on x by transaction Ti, the last accessor of x, meaning the last transaction that read or wrote x, is examined. If the last accessor is another transaction Tj, the algorithm verifies whether Tj had already committed or aborted before this write. If not, a violation is recorded, storing the position and the involved transactions. Both the last write and last access state for x are then updated to reflect this write. If the last writer or last accessor is the same transaction as the current operation, no intervening commit is required; the check applies only to different transactions.
After the pass completes, an explanation is constructed from the list of violations, describing at which position which transaction read or wrote which item while a previous accessor had not yet committed or aborted. The step-by-step explanation describes each relevant read and write, including first access, same-transaction access, or check against the last writer or last accessor, and concludes with a summary and final determination.

3.6 Two-Phase Locking (2PL) and Strict Two-Phase Locking (Strict 2PL)
Two-phase locking is a concurrency control protocol that ensures conflict-serializability through a disciplined approach to lock acquisition and release. A schedule follows the two-phase locking protocol if it can be produced by transactions that obey specific rules governing when locks may be acquired and released. The implementation provides validation for both Basic 2PL and Strict 2PL, determining whether a given schedule could have been generated under these protocols.
3.6.1 Lock compatibility rules
Both 2PL validations share the same underlying lock compatibility rules. For each data item x, the lock state tracks which transactions hold locks and in what mode. Read locks are shared: a read lock on x is granted only if no other transaction holds a write lock on x. Multiple transactions may simultaneously hold read locks on the same item. Write locks are exclusive: a write lock on x is granted only if no other transaction holds any lock, whether read or write, on that item. These rules ensure that conflicting operations are properly serialized through the locking mechanism.
3.6.2 Strict 2PL validation
A schedule follows Strict 2PL if it can be explained by a specific locking discipline: every data operation must be preceded by acquisition of the appropriate lock, and all locks held by a transaction must be released only at the moment that transaction commits or aborts. No early release of locks is permitted under Strict 2PL. The implementation validates Strict 2PL through a single left-to-right pass over the schedule, maintaining lock state for each data item and recording lock acquisitions and releases as they would occur under this discipline.
At each read or write operation, the algorithm checks whether the required lock can be granted according to the compatibility rules. If the lock can be granted, the appropriate lock acquisition is recorded, which may involve upgrading from a read lock to a write lock when a transaction writes an item it previously read. If the lock cannot be granted because another transaction holds a conflicting lock, a violation is recorded, and the schedule is marked as not following Strict 2PL. At each commit or abort, all locks held by that transaction are released. After the pass completes, the algorithm returns a boolean indicating whether the schedule follows Strict 2PL, along with a step-by-step explanation showing each lock acquisition, release, and any violations encountered.

3.6.3 Basic 2PL validation
Basic 2PL imposes a weaker set of constraints than Strict 2PL. A schedule follows Basic 2PL if there exists some assignment of lock acquisitions and releases that satisfies two conditions: first, every data operation must be preceded by acquisition of the appropriate lock; second, no transaction may acquire any lock after it has released its first lock. This is the classic two-phase rule: a growing phase where locks are only acquired, followed by a shrinking phase where locks are only released. Unlike Strict 2PL, early release of locks before commit or abort is permitted under Basic 2PL.
The implementation validates Basic 2PL by attempting to find a valid lock acquisition and release pattern that explains the schedule. Precomputed information records when transactions have finished using particular data items, allowing the algorithm to simulate early releases when other transactions are done with their locks. At each data operation, if the requesting transaction has already released any lock, a violation is immediately reported. If conflicting locks are held by other transactions, the algorithm checks whether those holders have finished using the item based on the precomputed information; if so, they are considered to have released early and their locks are removed. This process continues until either the lock can be granted or no further early releases are possible. If the lock still cannot be granted, the schedule cannot be explained under Basic 2PL and a violation is recorded.

3.6.4 Results and visualization
The implementation provides both validations and reports on their results together. If the Strict 2PL simulation encounters no conflicts, the schedule is reported as following both Strict 2PL and Basic 2PL, since Strict 2PL implies Basic 2PL. If the Strict 2PL simulation encounters a conflict, the schedule is reported as not following Strict 2PL, and the Basic 2PL validation determines whether it at least satisfies the weaker condition. This layered approach allows the system to distinguish between schedules that strictly follow the release-only-at-commit discipline and those that only satisfy the basic two-phase rule with early releases.
A separate feature provides a lock-history visualization. It does not perform validation. Instead, it assumes a schedule is legal and shows one possible sequence of lock and unlock operations that would be required under Strict 2PL to produce that schedule, acquiring read or write locks before each data operation and releasing all locks at commit or abort. This visualization is for explanatory purposes only, helping users see how Strict 2PL would execute the schedule; the actual validation determines whether the schedule could have been produced under the protocol.
3.7 Schedule generation
In addition to analyzing user-entered schedules, the system can generate schedules. Generation supports three modes: random interleaving of transaction operations, Basic 2PL, and Strict 2PL. In random mode, the implementation produces a valid interleaving of the operations of the given transactions without enforcing any locking discipline. In 2PL and Strict 2PL modes, the implementation simulates the corresponding protocol to produce a schedule that respects the same lock compatibility rules and release policy described in Section 3.6; the generator chooses which transaction executes next when several can proceed, and may abort a transaction to resolve deadlock. The generated schedule can subsequently be analyzed or validated using the same correctness and 2PL validation logic as for manually entered schedules. Discussion of limitations and possible extensions
5.1 Limitations
• Scope of correctness criteria:  The simulator focuses on conflict-serializability and the standard recoverability-related properties (RC, ACA, strict, rigorous). Other correctness notions such as view serializability, snapshot isolation, or multiversion concurrency control are not modeled.
• Locking and blocking behavior:  The 2PL and Strict 2PL analyses abstract away from actual blocking and waiting in a real lock manager. The validator checks compatibility and two-phase rules on a given schedule but does not simulate waiting queues, timeouts, or deadlock detection beyond simple abort heuristics in the schedule generator.
• Scale and performance: The algorithms are linear or near-linear in the size of the history and work well for course-sized examples. Very large industrial schedules (with thousands of transactions and operations) are not a primary target and have not been systematically optimized or benchmarked.
• Deployment: The web interface is designed for local use (Flask backend with a React frontend). The system does not currently include authentication, multi-user support, or persistent storage of histories and results.
5.2 Possible extensions
• Richer concurrency-control models: Extend the simulator to support additional protocols such as timestamp ordering, multiversion 2PL, and optimistic concurrency control, with corresponding schedule validators and visualizations.
• Deadlock detection and resolution: Enhance the locking subsystem to build a waits-for graph, detect deadlocks explicitly, and experiment with different resolution strategies (e.g., wait-die, wound-wait, or random victim selection) in generated schedules.
• Enhanced schedule generation: Add more configurable generation options (e.g., varying contention level, number of items, and abort patterns) and support reproducible scenarios via seeds exposed in the UI.
• User experience and visualization: Improve the web interface with richer graph layouts, filtering and zooming, side-by-side comparison of schedules, and export options (PDF/PNG) for inclusion in reports.
• Persistence and integration: Allow users to save and reload schedules, store analysis sessions, and potentially integrate the API into other teaching or testing tools used in the course.

Overall, while the current system already covers the core concepts of transaction scheduling and correctness, these extensions would make it more faithful to real-world database systems and more flexible as an educational and experimentation platform.
