TWO PHASE LOCKING
(each transaction must acquire all locks before releasing any)
When a scheduler receives an operation from a TM, it has 3 options:
 1. immediately schedule it by sending to DM (aggressive scheduler)
 2. delaying it by queuing it (conservative scheduler)
 3. reject it & cause trans to abort
Aggressive vs. Conservative schedulers ( trade-offs ) :
 Aggressive schedulers - avoid delaying operations 
 - give up the opportunity to reorder operations
 - may get stuck in a situation in which it has
 no hope of finishing the execution of all active
 transactions in a serializable fashion
 - risk rejecting operations of one or more transactions
 Conservative schedulers - avoid rejecting operations by 
 deliberately delaying operations
 - more leeway to reorder operations
 - less likely to get stuck in a situation
 where it has to rejection operations
 -- Information needed for conservative scheduler to anticipate
 future transactions
 READSET = set of items transactions will read
 WRITESET = set of items transactions will write 
 -- Very Conservative scheduler PREDECLARES READSETS and WRITESETS.
 TM begins by giving scheduler READSET/WRITESET
 which introduces the possibility of overstating READSET & WRITESET
2
 EXAMPLE : Procedure Fuzzy-readset begin
 Start;
 a := Read(x);
 if ( a > 0 ) then b := Read(y)
 else b := Read(z);
 COMMIT
 end
 ** transaction must predeclare the set of all data items it 
 might read ( {x,y,z} )
 ** overstating of readsets and writesets causes the scheduler
 to be even more conservative - delaying certain operations
 in anticipation of others that will never be issued. 
 
 -- Applications (situation dependent) determine scheduler choice 
 -- Advantage of one scheduler over the other

 LOCKING : Each item has associated with it a lock.
 2 cases in obtaining a lock ( before T1 can access x ):
 1. the scheduler examines x's lock and if no other
 transaction holds a lock on x the scheduler obtains one
 on behalf of T1.
 2. the scheduler examines x's lock and another transaction
 holds a lock on x so T1 waits until x becomes available.
3
 -- Ti obtaining a lock on item x is indicated by
 rli[x] - READ LOCK
 wli[x] - WRITE LOCK
 -- Ti releasing a lock on item x is indicated by
 rui[x] - RELEASE READ LOCK
 wui[x] - RELEASE WRITE LOCK
 -- 2 Locks ( pli[x] & qlj[y] ) CONFLICT if the following 3 conditions
 are true:
 1. x = y
 2. i != j
 3. operations p and q are conflicting types 
 -- Two locks conflict if they are on the same data items, they
 are issued by different transactions, and one or both of them
 are write locks.
BASIC 2PL rules: Actions of Scheduler are:
---------------
 1. Receives pi[x] from TM test pli[x] for conflicting qlj[x] already set.
 If so, delays pi[x], forcing Ti to wait, if not, sets pli[x],
 sends pi[x] to DM.
 ** prevents two transactions from concurrently accessing a data item in
 conflicting modes. Conflicting operations are scheduled in the same 
 order in which the corresponding locks are obtained.
 2. Once scheduler sets pli[x] for Ti, scheduler may not release that lock
 at least until after DM acknowledges it has processed lock’s
 corresponding operation pi[x].
 ** This ensures that the DM processes operations on a data item in the 
 order that the scheduler submits it. 
 3. Once scheduler has released lock for T, may not obtain another for T.
 ** This guarantees that all pairs of conflicting operations of two 
 transactions are scheduled in the same order. 
4
 e.g., T1: r1[x] -> w1[y] -> c1, T2: w2[x] -> w2[y] -> c2
H1 = 
rl1[x]r1[x]ru1[x]wl2[x]w2[x]wl2[y]w2[y]wu2[x]wu2[y]c2wl1[y]w1[y]wu1[y]c1
 H1 is not SR => r1[x] < w2[x] and w2[y] < w1[y] T1 -> T2 -> T1
Problem: T1 released lock on x before it obtained lock on y.
 Solution: Obey rules.
H2 = 
rl1[x]r1[x]wl1[y]w1[y]c1ru1[x]wu1[y]wl2[x]w2[x]wl2[y]w2[y]c2wu2[x]wu2[y]
 -- Stronger version of 2 & 3 above would be:
 2PL Scheduler may not release any Ti locks until DM has acknowledged
 processing all Ti's locks (close to strict 2PL).
CORRECTNESS of BASIC 2PL (guarantees serializability)
------------------------
 -- To prove correctness, need to characterize all 2PL histories.
 -- To characterize all 2PL histories, orderings of ops which must hold:
 1. Lock is obtained for each DB op before it executes
 ( ol[x] < o[x] ).
 2. Operation is executed by DM before lock released 
 ( o[x] < ou[x] ).
 Prop 3.1 H history produced by 2PL scheduler. If o[x] in C(H), then
 ol[x] & ou[x] in C(H) and
 ol[x] < o[x] < ou[x] . (rules 1 and 2)
 Prop 3.2 H history produced by 2PL scheduler.
 If pi[x] & qj[x] (i != j) are in conflict in C(H), then either:
 1. pui[x] < qlj[x] or
 2. quj[x] < pli[x]. (rule 1)
5
 Prop 3.3 H a complete history produced by 2PL scheduler.
 If pi[x] & qi[y] in C(H) then :
 pli[x] < qui[y]. (rule 3 - 2PL rule)
 Using 3.1, 3.2, 3.3 must show every 2PL history has acyclic SG.
 ---------------------------------
 -- If T1 -> T2 -> ... -> Tn in SG(H), then
 1. One of Ti's operations executes before and conflicts with
 one of Ti + 1's ( = j ) operations ( i.e., ri[x] -> wj[x] )
 2. Ti must have released some lock before Ti+1 set some lock
 3. Ti+2 set all locks before releasing any
 4. Ti+2 must have released some lock before Ti+3 set any 
 Thus one can conclude:
 1. T1 must have released some lock before Ti+3 set some lock
 2. Ti must have released some lock before Tj set some lock,
 for any pair (i,j) with i < j
 -- If SG(H) had a cycle T1 -> T2 -> T1, then
 Ti must have released some lock before Tj=Ti set some lock,
 but that contradicts the 2PL rule.
Lemma 3.4 Let H be a 2PL history, and suppose T1 -> Tj is in SG(H). Then for
some data item x and some conflicting operations pi[x] and qj[x] in H, 
pui[x] < plj[x].
Proof: Since Ti -> Tj, there must be a pair of conflicting operations pi[x] and
qj[x] such that pi[x] < qj[x]. By proposition 3.1
1. pli[x] < pi[x] < pui[x], and
2. qlj[x] < qj[x] < quj[x]. 
By proposition 3.2, either pui[x] < qlj[x] or quj[x] < pli[x]
But we would have qj[x] < pi[x], which contradicts pi[x] < qj[x]. Thus, 
Pui[x] < qlj[x] as desired.
6
Lemma 3.5 Let H be a 2PL history, and let T1 -> T2 -> … > Tn be a path in SG(H), 
where n > 1. Then for some data items x and y, and some operations pi[x] and qn[y] 
in H, pu1[x] < qln[y].
Proof: by induction on n.
Theorem 3.6: Every 2PL history H is serializable
Proof: Suppose by way of contradiction that SG(H) contains a cycle T1 -> T2 … -> 
Tn -> T1 where n > 1. By Lemma 3.5 for some data items x and y and some 
operations p1[x] and q1[y] in H, pui[x] < qli[y]. But this contradicts Proposition 3.3. 
Thus, SG(H) has no cycles and so, by serializability theorem H is serializable.
DEADLOCK 
A situation in which each transaction in a set of transactions
is blocked waiting for another transaction in the set, and 
therefore, none will become unblocked unless there is external intervention. 
 - Deadlock can happen in the presence of a 2PL scheduler.
 e.g., T1: r1[x]->w1[y]->c1 T2: w2[y]->w2[x]->c2
1. initially no one holds a lock
2. scheduler receives r1[x] and sets rl1[x]
3. scheduler receives w2[y] and set wl2[y]
4. scheduler receives w2[x], but does not set wl2[x] because it conflicts with 
rl1[x]. w2[x] is delayed
5. scheduler receives w1[y], but does not set wl1[y] because it conflicts with 
wl2[y]. w1[y] is delayed
- followed 2PL rules, but cannot continue => deadlock
7
 e.g., Lock Upgrade T1: r1[x]->w1[x]->c1 T2: r2[x]->w2[x]->c2 (reading 
 bank balances)
1. scheduler receives r1[x], and sets rl1[x]
2. scheduler receives r2[x], and sets rl2[x]
3. scheduler receives w1[x], but cannot set wl1[x] because it conflicts with 
rl2[x]. w1[x] is delayed
4. scheduler receives w2[x], but cannot set wl2[x] because it conflicts with 
rl1[x]. w2[x] is delayed
 - followed 2PL rules, but cannot continue. => deadlock
 DEADLOCK MANAGEMENT (3 approaches)
1. TIMEOUT: scheduler aborts any transaction waiting for more than t sec.
 Problems:
 - may abort trans not really deadlocked
 - if timeout period, t, is too large: actually deadlocked 
 transactions are not detected for long periods of time
 (& others waiting for deadlocked transactions may be 
 delayed so that they are aborted unnecessarily).
 - timeout period too short: many nondeadlocked trans aborted.
 ( ** tuning of t is important consideration ** )
2. DETECTION: Scheduler uses Wait-For-Graph (WFG)
 - Nodes of WFG are transactions
 - Ti -> Tj is an edge iff Ti is waiting for Tj to release some lock.
 - Deadlock iff cycle in WFG. 
 ( i.e. T1 -> T2 -> ... -> Tn -> T1 ) 
 - Scheduler adds edge when lock request is blocked & remove upon release.
 
8
 2 approaches for DETECTION of cycles in WFG
 1. Scheduler can check WFG every time request is delayed 
 (PREVENTION)
 - if ops frequently delayed but seldom deadlocked, bad approach.
 2. Scheduler can check periodically 
 (DETECTION-RESOLUTION) chooses VICTIM.
 - Common factors in victim selection:
 1. Amount of effort already expended in trans.
 2. Cost of aborting trans (depends on # of updates ...).
 3. Amount of effort to finish trans (predict based on type).
 4. Number of cycles containing the trans.
 5. Number of times trans has restarted (avoid "cyclic restart")
 3. PREVENTION (skip now, comes later)
 Wound-Wait: (Tr requests lock held by Th) (preemptive)
 If ts(Tr) < ts(Th) {Tr older} then Th aborted by Tr, else Tr waits.
 (older requester wounds, younger requester waits)
 Wait-Die: (Tr requests lock held by Th) (non-preemptive)
 If ts(Tr) < ts(Th) {Tr older} then Tr waits, else Tr dies.
 (older requester waits, younger requester dies)

---------------------
 - Scheduler usually implemented by Lock Mgr (LM) Trans Mgr (TM) pair.
 TM subsumes scheduler function ensuring lock set before op performed.
 TRANS → TM → LM → TM → DM → TM

Lock Manager maintains table of locks, queues for waiting trans,
------------
 - supports : Lock (tr-id, item-id, mode)
 Unlock (tr-id, item-id)
10
 - Lock setting and releasing is the bottleneck.
 - Rule of Thumb: should take < several hundred machine instr to
 set/release lock (including overhead for monitor call, supervisor
 call, context switch, etc).
 - Often optimized for special cases: set non-conflicting lock,
 release all trans locks at once.
 - LOCK TABLE:
 1. is usually a hashed file on item-id, x (contains queue header
 pointing to list of locks on x that have been set and list of trans
 waiting for x)
 2. usually contains only items actually locked.
 3. releasing trans read-locks should be basic operation (since TM
 releases them as soon as trans terminates)
 4. if scheduler is strict, TM releases write-locks as soon as DM ack's
 commit, so releasing write locks should be basic also.
 5. common practice is to link all trans read-locks and all write-locks
 in lock table.
 6. should be protected (only available to progs implement lock/unlock
 - make LM part of operating system itself,
 - monitor
 - Monitor = common data structure and set of operations on it
 used to control synchronization of concurrent processes
